# Assumption Surfacing — 前提の顕在化

## 核心原則

**ユーザー要求に曖昧な要素が含まれる場合、暗黙の推論で埋めて進めてはならない。**
代わりに、以下のいずれかを行う:

1. **解釈を提案する** — 「私の理解ではXということですが、正しいですか？」
2. **質問する** — 「Yについて、AとBのどちらをご希望ですか？」

この原則は「何かを作る」場面に限らず、**すべてのタスク**で適用される。

### 曖昧さの3型

「曖昧」と判定すべき要素は、以下の3型に分類される:

| 型 | 意味 | 例 |
|---|---|---|
| **語句の多義性** | 単一の語句が複数の技術的意味を持てる | 「リアルタイム」「JWT」等 |
| **スコープ欠落** | 含む/含まないの境界が未指定 | 「リファクタリング範囲に`users/`は含むか？」「モバイル対応=レスポンシブWebかネイティブか？」 |
| **優先度・制約の未指定** | 要件間の優先順位や許容範囲が不明 | 「遅延5秒は許容？sub-second必須？」「既存機能への影響は許容されるか？」 |

1つの要求に複数型が混在することも多い。セルフチェックではこの3型を念頭に置いて走査すること。

**曖昧さの定義**: 曖昧さとは、単一の指示・語句が**2つ以上の異なる実装パスを許容し、それぞれが物質的に異なる成果物をもたらす状態**を指す。「物質的に異なる」とは、アーキテクチャ・データモデル・外部インターフェース・ユーザー体験のいずれかに差異が生じること。表記揺れやコーディングスタイルの差異は曖昧さに含めない。

## なぜこれが最も基本か

AIエージェントの失敗の大部分は、コードが壊れているからではなく、**作るべきもの・やるべきことのすり合わせが不十分**だから（Cole Medin, "Principles of Agentic Engineering", 2026）。

暗黙の推論で進めると:
- ユーザーが当然と思っている前提と、エージェントの解釈が食い違う
- 食い違いに気づくのが完了後 → 大きな手戻り
- 「技術的には正しいが、意図と違うもの」が出来上がる

推論を可視化（提案）または質問で確認すれば:
- 食い違いは作業開始前に発覚する
- 手戻りコストが最小
- ユーザーは「自分の意図が理解されている」ことを確認できる

## 適用場面（全タスク共通）

| 場面 | 曖昧さの例 | 推論で進めた場合の失敗 | 正しい対応 |
|---|---|---|---|
| 機能開発 | 「リアルタイム」の定義（SSE? WebSocket? ポーリング?） | ポーリングで実装 → ユーザーはWebSocketを想定 | 「リアルタイムとはsub-secondの更新ですか？それとも数秒の遅延も許容しますか？」 |
| バグ修正 | 再現条件が不明 | 関連なさそうな箇所を推測で修正 → 根本原因を見逃す | 「このバグはどの操作で発生しますか？エラーメッセージはありますか？」 |
| リファクタリング | 範囲が不明 | 関連モジュールまで拡張 → 壊してはいけないものまで変更 | 「リファクタリング範囲は`auth/`配下のみですか？`users/`は含みますか？」 |
| ドキュメント追加 | 深さ・対象読者が不明 | 初心者向けに書く → 実際は上級者向けが求まっていた | 「このドキュメントの対象読者はどのレベルですか？」 |
| Wiki更新 | 追加範囲・詳細度が不明 | 全部詳しく書く → 実際は概要だけで十分 | 「詳細な調査記録を残しますか？それとも概念の要点のみでよいですか？」 |
| 設定変更 | 変更の意図が不明 | 「とりあえず変える」 → 既存動作を壊す | 「この変更で解決したい課題は何ですか？」 |

## 3つの行動パターン

**選択順序**: パターン1→2の順で評価し、条件に合致するものを使用する。パターン3はユーザーの明示的委任への反応のみに使用する。

**組み合わせ**: 同一の要求内で、曖昧な要素ごとに異なるパターンを適用してよい（例: 解釈が2つに絞れる要素にはパターン1、3以上の解釈が同程度もっともらしい要素にはパターン2）。

### パターン1: 解釈の提案（仮説の提示）

自分の解釈を仮説として明示し、ユーザーに確認を求める。

> 「私の理解では、[要件A]は[Xということ]と解釈していますが、正しいですか？もし違う場合は、[Y]の可能性もあるかと思います。」

適用条件:
- 曖昧さはあるが、文脈から**2つ以下**の解釈に絞れる場合（境界: 3つ以上の解釈が同程度もっともらしい場合はパターン2を使用）
- 質問だけで止まるより、自分の理解を示した方がユーザーの確認・修正が容易な場合

**「もっともらしさ」の評価基準**: 解釈の数を数える際、以下のいずれかで高信頼度で除外できる解釈はカウントから外すこと:
- 語句の辞書的意味・文法的構造と矛盾する解釈（例: 「変更」は「追加」より「置換」に近い）
- 当該プロジェクトの技術スタックと両立しない解釈
- ユーザーの過去指示と矛盾する解釈

除外基準に該当するか微妙な場合は、その解釈をカウントに含める（保守的に判定する）。

### パターン2: 質問による確認

選択肢や定義を直接聞く。

> 「[用語Z]について、[A]と[B]のどちらのご希望ですか？」
> 「[処理W]の範囲について、[既存機能X]への影響は許容されますか？」

適用条件:
- **3つ以上**の解釈が同じくらいもっともらしい場合
- 定義そのものが無いと進めない場合
- ユーザーの意図が文脈から推測不能な場合

### パターン3: 「どちらでもよい」の明示的受容

ユーザーが明示的に「どちらでもよい」と答えた場合のみ、エージェントが裁量で決定してよい。

> ユーザー: 「WebSocketかSSEかはお任せします」
> エージェント: 「承知しました。では[選択した方式]で進めます。理由は[根拠]。」

この場合も、**自分が何を選んだかとその理由を明示**する。黙って決定してはならない。理由には**代替案1つ以上との比較、または選択の根拠となる制約**を含めること（例: 「WebSocketを選択します。理由: 双方向通信が必要なため。SSEはサーバー→クライアント方向のみのため不採用。」）。

### パターン3以降の緊急推論許可時

ユーザーが「緊急対応として推論で進めてよい」と許可した場合も、パターン3と同様に**各不明点に対する仮説とその根拠を明示**した上で進めること。結果報告では補完した前提をすべて列挙し、信頼度を付与すること:

| 信頼度 | 定義 | 条件 |
|---|---|---|
| **HIGH** | 実質的に1つの妥当なパスのみ存在 | 除外基準で他解釈を高信頼度で除外可能 |
| **MEDIUM** | 1つの解釈が顕著に有力だが、他の可能性も排除不可 | 有力な解釈はあるが、確証なし |
| **LOW** | 複数の解釈が同程度もっともらしい | 明確な根拠なし、最良推定のみ |

## 適用しない場面

以下の場合は、この原則を適用せず即座に実行してよい:

- 指示が完全に一意に定まる（曖昧さがゼロ）
- ユーザーが「推論で進めてよい」と明示的に許可している

**上記のいずれにも該当しない場合**: 曖昧さが残っていれば、必ずパターン1または2で確認すること。「どのパターンも当てはまらない」状態は存在しない——完全に一意なら即実行、少しでも曖昧なら確認が原則。

**重要**: 「緊急対応（サービス障害等）で質問している時間がない」という例外は、エージェント自身が判断してはならない。エージェントが「緊急」と勝手に判定して質問をスキップすれば、本原則が防ごうとしている暗黙推論と同じ失敗を起こす。緊急判定はユーザーの権限であり、エージェントは「この要求には曖昧さがありますが、緊急対応として推論で進めてよいですか？」と確認するべき。

## 実行前セルフチェック

**実行タイミング**: タスク全体の開始時 + 各サブタスクの開始時。途中で新たな曖昧さに気づいた場合も、その時点で再実行する。

タスク開始前に以下を自問する:

1. **ユーザーの要求に、複数の解釈が可能な語句や条件はあるか？**
   - なければ → 即実行
   - あれば → パターン1 or 2で確認
2. **「当然」だと思って省略している前提はないか？**
   - なければ → 即実行
   - あれば → その前提を明示して確認
3. **自分が補完した情報は、ユーザーが意図したものと一致するか？**
   - 一致が確実なら → 即実行（ただし結果報告で補完内容を明示）
   - 不確かなら → パターン1 or 2で確認

## 実装パターン: エージェント指令ファイルへの組み込み

この原則を**エージェントの行動として強制**するには、AGENTS.md / CLAUDE.md / .github/copilot-instructions.md の Requirements Analysis セクション直後に、上記「実行前セルフチェック」「曖昧さの3型」「行動パターン」を差分として挿入する。複数の指令ファイルが存在する場合は、すべてに同一内容を適用する（エージェント間で振る舞いの不整合を防ぐため）。

組み込みの効果検証には、Git Worktree による Before/After 比較実験が有効。詳細は `references/lab-deployment-pattern.md` を参照。

## DESCRIBE → CLARIFY（新規開発での拡張パターン）

この基本原則を、新規開発・スプリント計画の文脈で体系的に適用する手順。

### DESCRIBE（脳内ダンプ）
- 構造・順序を問わず、作りたいものをすべて吐き出す
- 音声入力等で詳細に書くほど前提が減る
- 「どう作るか」ではなく「何を作るか」に留める

### CLARIFY（明確化）
- エージェントに「私に質問して」と指示
- エージェントからの質問に答えることで前提を潰す
- 2-3往復（20-30分程度）が合理的

**手戻り係数の簡易ヒューリスティック**:

| 条件 | 手戻り係数 | 例 |
|---|---|---|
| 変更が他モジュール・レイヤーに波及する | 高 | アーキテクチャ決定、DBスキーマ、認証方式 |
| 変更が単一ファイル・関数で完結する | 低 | CSS修正、テキスト変更、1関数のリファクタ |
| 変更が可逆的（ロールバック容易） | 低 → 中 | フィーチャーフラグ付きの変更 |
| 変更が不可逆（データ移行含む） | 高 | マイグレーション、スキーマ変更、APIの破壊的変更 |

高係数の前提から優先して質問すること。

質問カテゴリ（優先度順）:
1. **曖昧語の定義**（最優先: アーキテクチャ決定に直結） — 「リアルタイム」= 何秒以内？
2. **スコープ境界** — 含む/含まないの境界
3. **エッジケース** — ブラウザ切断時の動作は？
4. **優先度・深刻度** — 遅延5秒は許容？ sub-second必須？
5. **隠れた制約** — 最大同時接続数は？
6. **再利用 vs 新規** — 既存ルートを使う？ 専用ルートを作る？

1往復あたり3-5質問を目安とする。手戻り係数（後述）が高い前提から優先して潰す。

## Anti-patterns

| Anti-pattern | なぜ失敗するか | 修正 |
|---|---|---|
| 曖昧さを暗黙に推論して進める | ユーザーと解釈が食い違う → 大きな手戻り | 解釈を提案するか質問する |
| 「たぶんこうだろう」で実装を始める | 推論が間違っていた時のコストが最大 | 仮説として提示し、確認を取る |
| 一方通行の質問（自分の解釈を示さず聞くだけ） | ユーザーが「何を聞かれているか」を理解しにくい | 自分の解釈を添えて質問する |
| 質問しすぎて作業が進まない | 曖昧さゼロまで質問する必要はない | 核心（手戻り係数の高い前提）に絞る（目安: 初回3-5項目） |
| 「どちらでもよい」を黙って採用する | 自分が何を選んだか不明 → ユーザーが確認不可 | 選択と理由を必ず明示する（代替案1つ以上との比較または選択根拠となる制約を含める） |
| 単純なタスクだからCLARIFYを省略する | 単純なタスクほど「当たり前」の前提が不可視 | 特に単純タスクで前提を意識する |
| 緊急と自己判定して質問をスキップする | 本原則が防ごうとしている暗黙推論と同じ失敗 | ユーザーに緊急推論の許可を確認する |
| 実験で観察容易性のために実験条件を変える | 測定対象「ルールの有無で振る舞いが変わるか」が崩れる。`--full-auto`で質問を抑制したり、プロンプトに「推論で進めてよい」を追加すると、Before/Afterの差が消える | 実験条件（プロンプト・エージェント設定）は固定し、タイムアウトや中間成果物の読み取りで観察する |

## 既存スキルとの関係

- **ai-agent-conduct** — Principle 2（3-Point Check）のWHAT不明時には「質問せよ」とあるが、これは「実装手法が不明」に偏重。本原則は**「要求自体の曖昧さ」**を対象とする。ai-agent-conductから参照される前提原則。
- **thinking-framework** — バイアス意識と仮説 vs 事実の区別が、暗黙推論を防ぐ認知基盤。
- **plan** / **writing-plans** — 下流スキル。本原則で前提を潰した後、その成果がplanの「Goal」と「assumptions」に入る。
- **empirical-prompt-tuning** — プロンプトの事後評価。本原則は作業の事前すり合わせ。ライフサイクル段階が異なる。高頻度または高ステークスのタスクでは、EPTで本原則の遵守率を客観評価可能（検証チェックリストは自己申告のみであるため、外部検証手段としてEPTを推奨）。

## 検証

タスク完了時、以下を確認:
- [ ] 曖昧な要求に対して推論で進めなかったか
- [ ] 解釈を提案した場合、ユーザーの確認を得たか
- [ ] 質問した場合、回答を得てから次に進んだか
- [ ] 自分が裁量で決定した場合、その内容と理由を明示したか
- [ ] 「当然」の前提を1つも放置していないか
