# Contributing

## Quick Start

1. Clone the repository.
2. Open it in VS Code.
3. Run Dev Containers: Reopen in Container.
4. Wait for `.devcontainer/post-create.sh` to refresh pinned versions in `package.json`, `pyproject.toml`, and `.devcontainer/devcontainer.json`, sync project dependencies, and install AI CLI tools.
5. Verify the baseline locally:

```bash
npm ci
uv sync --dev
npm run lint
npm run test:shells
npm run test
```

These commands match the smoke CI workflow in [.github/workflows/ci.yml](.github/workflows/ci.yml).

The Dev Container defaults to `en_US.UTF-8` and `Etc/UTC`. Treat those values as baseline defaults rather than universal requirements for every derived template.

## Worktree Workflow

This repository assumes that multiple worktrees may exist under `.worktree/` while `/home/vscode` stays shared.

After you add a new worktree, run the dependency sync commands inside that worktree because `postCreateCommand` only runs when the Dev Container is created:

```bash
./.devcontainer/scripts/update-tools.sh node-deps
./.devcontainer/scripts/update-tools.sh python-deps
```

If you are starting from the repository root, a typical flow is:

```bash
mkdir -p .worktree
git worktree add .worktree/feature-a -b feature-a
cd .worktree/feature-a
./.devcontainer/scripts/update-tools.sh node-deps python-deps
```

## Tooling And Updates

Use `./.devcontainer/scripts/update-tools.sh --list` to inspect the supported update targets.

Common commands:

```bash
./.devcontainer/scripts/update-tools.sh node-deps
./.devcontainer/scripts/update-tools.sh node-deps-check
./.devcontainer/scripts/update-tools.sh python-deps
./.devcontainer/scripts/update-tools.sh python-deps-check
./.devcontainer/scripts/update-tools.sh uv claude-code
./.devcontainer/scripts/check-package-updates.sh
./.devcontainer/scripts/check-python-package-updates.sh
./.devcontainer/scripts/update-tools.sh --versions
```

`check-package-updates.sh` reports package.json range updates, sync-only dependency drift, and `npm audit`
findings separately. It exits with status `1` when something needs attention and `2` on usage or runtime errors.

`check-python-package-updates.sh` reports Python specifier updates, `uv.lock` refresh candidates,
environment sync drift, and `uv audit` findings separately. It exits with status `1` when something needs
attention and `2` on usage or runtime errors.

`node-deps` refreshes `package.json` dependency pins, refreshes `package-lock.json`, and then runs `npm ci`.

`python-deps` refreshes `pyproject.toml` dependency pins, refreshes `uv.lock`, and then runs `uv sync --dev`.

`update-tools.sh` and `update-tools.sh --all` update only the default tool set. Run `node-deps` and
`python-deps` explicitly when you want to refresh workspace dependencies.

Manifest pin refreshes stay within the current major version by default. For Python requirements, an existing
stricter upper bound is preserved.

`node`, `npm`, and `python` also refresh the pinned runtime versions used by the Dev Container baseline on the
next rebuild by updating `.devcontainer/devcontainer.json` and `package.json` where appropriate.

Because those commands rewrite pins and lockfiles, review and commit the resulting `package.json`,
`pyproject.toml`, `package-lock.json`, `uv.lock`, and `.devcontainer/devcontainer.json` changes after a
rebuild or manual sync if you want CI and collaborators to use the same versions.

Existing shell scripts under `.devcontainer/` are expected to keep both a shebang and executable permissions.

## AI-Generated Code

This repository installs AI coding agent CLIs as optional tools. Code generated by AI agents is held to
the same quality bar as human-authored code:

- All CI checks (`npm run lint`, `npm run test:shells`, `npm run test`) must pass regardless of authorship.
- AI-generated changes require the same human review as any other pull request.
- Do not copy third-party code verbatim from AI output without confirming its license is compatible with
  this repository's MIT license.
- The PR checklist in `.github/pull_request_template.md` includes explicit items for these requirements.

## VS Code Extensions

The repository uses two layers for extensions:

- `.devcontainer/devcontainer.json` contains the extensions that should be available automatically inside the container baseline.
- `.vscode/extensions.json` contains the broader recommended set for contributors.

When adding a new extension, use this rule:

- Add it to `.devcontainer/devcontainer.json` only if the container experience depends on it being present by default.
- Add it to `.vscode/extensions.json` when it is useful for contributors but not required for the baseline to function.

## Home Volume

`/home/vscode` is mounted from the fixed Docker volume `devcontainer-home`.

That means:

- CLI sign-in state and caches survive rebuilds.
- Multiple worktrees for this repository share the same Home state.
- Separate clones or separate repositories on the same host can also collide if they reuse the same volume name.

If the Home state becomes inconsistent, remove the volume on the host and reopen the container:

```bash
docker volume rm devcontainer-home
```

## Troubleshooting

If a worktree is missing dependencies, rerun the local sync commands:

```bash
./.devcontainer/scripts/update-tools.sh node-deps python-deps
```

If a shell script fails with `Permission denied`, verify the executable bit:

```bash
find .devcontainer -type f -name '*.sh' -exec test -x {} \; -print
```

If Ollama does not respond after container start, inspect the logs:

```bash
printf 'OLLAMA_HOST=%s\n' "${OLLAMA_HOST:-<unset>}"
tail -n 20 ~/.local/state/ollama/post-start.log
tail -n 20 ~/.local/state/ollama/server.log
```

For the default local mode, `OLLAMA_HOST` should be `127.0.0.1:11434`. If it still shows
`host.docker.internal:11434`, the current shell is still carrying an older environment. Open a fresh terminal after
rebuild or rebuild the container again before debugging further.
