# アジャイル思考サイクル

> 作成日: 2025-12-02
>
> 本ドキュメントは、思考プロセスを反復的に回し、変化に適応しながら成果を出すための「アジャイル思考」を解説します。
> [コンサルティング思考フレームワーク](./consulting-thinking-framework.md)で学んだ思考の型を、実際にどう「回す」かに焦点を当てています。

---

## 1. 概要

### 1.1 アジャイル思考とは

アジャイル思考とは、**完璧な計画を立ててから実行するのではなく、小さく素早く試して、学びながら軌道修正していく**思考・行動様式です。

```
【従来型（ウォーターフォール）】
計画 ────────→ 実行 ────────→ 結果
（最初に全部決める）      （最後にわかる）

【アジャイル型】
計画 → 実行 → 学習 → 計画 → 実行 → 学習 → ...
（小さく回し続ける）
```

### 1.2 なぜアジャイルが必要か

| 状況 | 従来型の問題 | アジャイルの利点 |
|------|--------------|------------------|
| **不確実性が高い** | 最初の計画が外れると全部やり直し | 早期に軌道修正できる |
| **変化が速い** | 計画中に前提が変わる | 変化を取り込みながら進める |
| **正解がわからない** | 机上で考えても答えが出ない | 試して学ぶことで答えに近づく |

### 1.3 3つのドキュメントの関係

本ドキュメント（アジャイル思考サイクル）、[コンサルティング思考フレームワーク](./consulting-thinking-framework.md)、[メタ思考](./meta-thinking.md)は以下の関係にあります。

```
┌─────────────────────────────────────────────────────────────┐
│              アジャイル（反復プロセス全体）                  │
│              本ドキュメント                                  │
│                Sprint 1 → Sprint 2 → Sprint 3 → ...        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   各粒度（Epic / Story / Task）ごとに：                    │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐   │
│   │  メタ認知/メタ思考（常時稼働）                      │   │
│   │  「今の思考・判断は正しいか？」を問い続ける        │   │
│   │  → 詳細は meta-thinking.md                         │   │
│   ├─────────────────────────────────────────────────────┤   │
│   │  思考フレームワーク（1回の思考プロセス）           │   │
│   │  論点→仮説→戦略→戦術                              │   │
│   │  ＋ 4層構造（素養/技術/リサーチ/武器）            │   │
│   │  → 詳細は consulting-thinking-framework.md        │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                             │
│   → 粒度を下げる（解像度を上げる）ごとに繰り返す         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.4 各ドキュメントの役割

| ドキュメント | 役割 | 問い |
|--------------|------|------|
| **本ドキュメント（アジャイル）** | 反復プロセス全体の回し方 | 「どうサイクルを回すか？」 |
| **meta-thinking.md** | 各粒度での監視・調整 | 「今のプロセス自体は正しいか？」 |
| **consulting-thinking-framework.md** | 1サイクルの中身 | 「1回の思考で何をどう使うか？」 |

## 2. 代表的なサイクルモデル

### 2.1 PDCA サイクル

最も基本的な改善サイクル。**計画重視**で、比較的安定した環境に向く。

```
    Plan（計画）
         ↓
    Do（実行）
         ↓
    Check（評価）
         ↓
    Act（改善）
         ↓
    （Plan に戻る）
```

| フェーズ | 内容 | 例（テスト勉強） |
|----------|------|------------------|
| **Plan** | 目標と計画を立てる | 「数学で80点取るために、毎日2時間勉強する」 |
| **Do** | 計画を実行する | 実際に勉強する |
| **Check** | 結果を評価する | 模試を受けて65点だった |
| **Act** | 改善策を考える | 「応用問題が弱いので、問題集を追加する」 |

### 2.2 OODA ループ

**観察・状況判断重視**で、変化が速く不確実な環境に向く。軍事戦略から生まれた概念。

```
    Observe（観察）
         ↓
    Orient（状況判断）
         ↓
    Decide（意思決定）
         ↓
    Act（行動）
         ↓
    （Observe に戻る）
```

| フェーズ | 内容 | 例（サッカーの試合中） |
|----------|------|------------------------|
| **Observe** | 状況を観察する | 「相手DFが前がかりだ」 |
| **Orient** | 状況を解釈・判断する | 「裏にスペースがある」 |
| **Decide** | 行動を決める | 「スルーパスを出そう」 |
| **Act** | 行動する | パスを出す |

### 2.3 PDCA と OODA の使い分け

| 観点 | PDCA | OODA |
|------|------|------|
| **適した環境** | 安定・予測可能 | 変化・不確実 |
| **重視するもの** | 計画の精度 | 状況への適応速度 |
| **サイクル速度** | 比較的ゆっくり | 速い |
| **例** | 業務改善、品質管理 | 競争環境、危機対応 |

## 3. アジャイル開発の原則（ソフトウェア開発から学ぶ）

ソフトウェア開発で生まれたアジャイルの原則は、思考全般に応用できます。

### 3.1 アジャイル宣言の4つの価値

| 従来重視されていたもの | → | アジャイルで重視するもの |
|------------------------|---|--------------------------|
| プロセスやツール | → | **個人と対話** |
| 包括的なドキュメント | → | **動くソフトウェア（成果物）** |
| 契約交渉 | → | **顧客との協調** |
| 計画に従うこと | → | **変化への対応** |

### 3.2 思考への応用

| アジャイル原則 | 思考への応用 |
|----------------|--------------|
| **動くものを早く作る** | 完璧な分析より、まず仮説を検証する |
| **変化を歓迎する** | 前提が変わったら、計画を柔軟に修正する |
| **短いサイクルで回す** | 大きな賭けより、小さな実験を繰り返す |
| **振り返りを重視する** | 定期的に「やり方自体」を見直す |

## 4. 仮説検証ループ

### 4.1 基本構造

アジャイル思考の核心は「仮説検証ループ」です。

```
┌─────────────────────────────────────────────┐
│              仮説検証ループ                 │
│                                             │
│    仮説を立てる                             │
│         ↓                                   │
│    検証方法を設計する                       │
│         ↓                                   │
│    最小限の実験で検証する                   │
│         ↓                                   │
│    結果から学ぶ                             │
│         ↓                                   │
│    仮説を更新する（または破棄する）         │
│         ↓                                   │
│    （最初に戻る）                           │
└─────────────────────────────────────────────┘
```

### 4.2 MVP（Minimum Viable Product）の考え方

**MVP = 検証に必要な最小限の成果物**

完璧なものを作ってから検証するのではなく、**検証できる最小限のもの**をまず作る。

| 悪い例 | 良い例（MVP） |
|--------|---------------|
| 「完璧なアプリを1年かけて作ってからリリース」 | 「コア機能だけのβ版を1ヶ月で出してユーザーの反応を見る」 |
| 「100ページの事業計画書を書いてから提案」 | 「1枚の仮説シートで上司に壁打ちしてもらう」 |
| 「全科目を完璧に勉強してからテスト」 | 「1章だけ勉強して小テストで理解度を確認」 |

### 4.3 ピボット（方向転換）

検証の結果、仮説が間違っていたら**ピボット（方向転換）**する。

- 失敗ではなく「学び」として捉える
- 早く失敗するほど、早く正解に近づける
- **「失敗のコスト」を最小化**することが重要

## 5. スプリント（時間で区切る）

### 5.1 スプリントとは

**スプリント = 固定された短い期間（1〜4週間）で区切った作業単位**

時間で区切ることで、強制的に「計画→実行→振り返り」のサイクルを回します。

```
スプリント1    スプリント2    スプリント3
[2週間]        [2週間]        [2週間]
計画→実行→    計画→実行→    計画→実行→
振り返り       振り返り       振り返り
```

### 5.2 スプリントの構成要素

| 要素 | 内容 | タイミング |
|------|------|------------|
| **スプリント計画** | 今回のスプリントで何をやるか決める | スプリント開始時 |
| **デイリースタンドアップ** | 毎日の進捗確認（15分程度） | 毎日 |
| **スプリントレビュー** | 成果物のレビュー、フィードバック | スプリント終了時 |
| **レトロスペクティブ（振り返り）** | やり方自体の改善点を議論 | スプリント終了時 |

### 5.3 個人での活用例

| 場面 | スプリント設計 |
|------|----------------|
| **資格勉強** | 1週間単位で「今週やる範囲」を決め、週末に振り返り |
| **ダイエット** | 2週間単位で「今期の目標体重と行動」を決め、結果を見て次の2週間を調整 |
| **プロジェクト** | 1ヶ月単位でマイルストーンを設定し、月末に進捗と計画を見直し |

## 6. 粒度（Epic / Story / Task）とメタ認知の関係

### 6.1 各粒度でメタ認知＋思考フレームワークを適用

アジャイル開発では、Epic → Story → Task と粒度を下げて（解像度を上げて）いきます。
**各粒度ごとに、メタ認知と思考フレームワークを適用**します。

```
Epic（大きな目標）
│
│  ┌─────────────────────────────────────────┐
│  │ メタ認知：「このEpicの設定は正しいか？」│
│  │ 思考FW：論点→仮説→戦略→戦術           │
│  └─────────────────────────────────────────┘
│
├──→ Story（機能単位）
│    │
│    │  ┌─────────────────────────────────────────┐
│    │  │ メタ認知：「このStoryの優先度は正しいか？」│
│    │  │ 思考FW：論点→仮説→戦略→戦術           │
│    │  └─────────────────────────────────────────┘
│    │
│    ├──→ Task（作業単位）
│         │
│         │  ┌─────────────────────────────────────────┐
│         │  │ メタ認知：「このTaskのやり方は正しいか？」│
│         │  │ 思考FW：論点→仮説→（簡易な）戦術       │
│         │  └─────────────────────────────────────────┘
```

### 6.2 粒度ごとのメタ認知の問い

| 粒度 | メタ認知の問い | 思考フレームワークの適用 |
|------|----------------|--------------------------|
| **Epic** | 「この目標自体が正しいか？」「市場環境は変わっていないか？」 | フル適用（論点→仮説→戦略→戦術） |
| **Story** | 「この機能は本当に必要か？」「優先度は正しいか？」 | フル適用 |
| **Task** | 「このやり方で効率的か？」「もっと良い方法はないか？」 | 簡易適用（論点→戦術） |

### 6.3 粒度を下げる際の注意点

- **上位の論点を常に意識する** - Taskに没頭してEpicの目的を見失わない
- **定期的に上位に戻る** - Sprint Reviewで「Epicに対して進捗しているか」を確認
- **メタ認知で監視する** - 「今、どの粒度にいるか」を常に意識する

## 7. 思考フレームワークとの統合

### 7.1 1スプリントの中で思考フレームワークを回す

```
スプリント開始
      ↓
┌─────────────────────────────────────────────┐
│  メタ認知：「正しい問いを設定しているか？」│
├─────────────────────────────────────────────┤
│  論点思考：今回のスプリントで解く問いは？   │
│  仮説思考：答えの仮説は？                   │
│  戦略思考：どこに集中する？                 │
│  戦術思考：具体的に何をする？               │
└─────────────────────────────────────────────┘
      ↓
    実行
      ↓
    検証・学習
      ↓
┌─────────────────────────────────────────────┐
│  メタ認知：「このサイクルの回し方は正しかったか？」│
└─────────────────────────────────────────────┘
      ↓
スプリント終了 → 振り返り → 次のスプリントへ
```

### 7.2 スプリントごとに論点・仮説を更新する

| スプリント | 論点 | 仮説 | 結果 | 学び |
|------------|------|------|------|------|
| **1** | 売上が落ちている原因は？ | 新規顧客の減少が主因 | 検証の結果、既存顧客の離脱が主因だった | 論点は合っていたが仮説が違った |
| **2** | 既存顧客が離脱する原因は？ | 価格が高いから | 検証の結果、サポート品質への不満だった | 仮説を修正 |
| **3** | サポート品質をどう改善するか？ | 対応速度を上げる | 対応速度を上げたら満足度が向上した | 仮説が正しかった。施策を拡大 |

## 8. 振り返り（レトロスペクティブ）の技法

### 8.1 KPT（Keep / Problem / Try）

最もシンプルな振り返りフレームワーク。

| 項目 | 問い | 例 |
|------|------|-----|
| **Keep** | 続けるべきことは？ | 「毎朝30分の勉強時間は効果的だった」 |
| **Problem** | 問題だったことは？ | 「夜の勉強は集中できなかった」 |
| **Try** | 次に試すことは？ | 「夜は復習に切り替えて、新しい内容は朝にやる」 |

### 8.2 YWT（やったこと / わかったこと / 次にやること）

日本発の振り返りフレームワーク。

| 項目 | 内容 |
|------|------|
| **Y（やったこと）** | 実際に行ったこと |
| **W（わかったこと）** | やってみて気づいたこと・学び |
| **T（次にやること）** | 次のアクション |

### 8.3 振り返りの習慣化

| 頻度 | 対象 | 例 |
|------|------|-----|
| **毎日** | 今日の行動 | 日報、日記 |
| **毎週** | 1週間の成果 | 週次レビュー |
| **毎月** | 月間の進捗 | 月次振り返り |
| **四半期** | 大きな方向性 | 四半期レビュー |

## 9. アジャイル思考のアンチパターン（よくある失敗）

| アンチパターン | 問題 | 対策 |
|----------------|------|------|
| **計画しすぎ** | 最初に完璧な計画を立てようとして動けない | まず小さく始める |
| **振り返りをしない** | 同じ失敗を繰り返す | 振り返りを習慣化する |
| **学びを活かさない** | 検証しても次に反映されない | 学びを明文化し、次の計画に組み込む |
| **サイクルが長すぎ** | 変化に対応できない | サイクルを短くする |
| **完璧を求めすぎ** | MVPを出せない | 「検証に必要な最小限」を意識する |
| **メタ認知の欠如** | 上位の目的を見失ってTaskに没頭 | 定期的に「正しい問いか？」を問う |

## 10. まとめ

### 10.1 アジャイル思考の本質

1. **完璧より「まず動く」** - 考えすぎる前に小さく試す
2. **計画より「適応」** - 変化を前提に柔軟に対応
3. **成功より「学び」** - 失敗も学びとして次に活かす
4. **振り返りを習慣化** - やり方自体を継続的に改善
5. **各粒度でメタ認知** - Epic/Story/Taskごとに「正しいか？」を問う
